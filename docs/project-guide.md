# ARProperty Project Guide

이 문서는 팀이 현재 구현 기준으로 참조하는 핵심 기준 문서입니다. 장문의 조사 자료나 과거 초안은 `docs/archive/`에 보존하고, 현재 팀이 합의한 방향만 이 문서에 유지합니다.

## 1. 프로젝트 요약

- 대상: 구미시 아파트 현장 탐색
- 사용자 가치: 현장에서 보고 있는 아파트에 대해 가격, 건축 정보, 생활 인프라를 한 번에 확인
- 핵심 경험: `카메라로 본다 -> 동을 고른다 -> 거래/상세/생활 점수를 확인한다`

한 문장 정의:

> 구미시에서 사용자가 스마트폰 카메라로 아파트 동을 비추면, 해당 건물의 실거래가와 건축 정보, 생활 인프라 점수를 AR과 지도에서 함께 보여주는 현장형 부동산 탐색 서비스

## 2. 현재 저장소 기준

현재 저장소는 구현 뼈대 단계입니다.

- `android/`: Compose 기반 Android Studio scaffold 존재
- `backend/`: Spring Boot 설정, Dockerfile, 환경설정, DB 스키마는 존재
- `backend/src/main/java`: 애플리케이션 소스 미커밋
- `data/`: 수집 스크립트와 매핑 파일 존재
- `docs/`: 현재 기준 문서 정리 완료

이 문서는 “이미 구현된 기능 설명서”가 아니라, 현재 팀이 같은 방향으로 움직이기 위한 기준 문서입니다.

## 3. 범위와 우선순위

### 이번 프로젝트에서 하는 것

- Android Kotlin 기반 앱
- Spring Boot 기반 REST API
- PostgreSQL + PostGIS 기반 단지/동/거래 데이터 저장
- 구미시 아파트 데이터 적재
- 생활 인프라 점수 계산과 비교

### 이번 프로젝트에서 하지 않는 것

- iOS 동시 지원
- 세대(호수) 단위 식별
- 전국 단위 확장
- 문서 기준보다 앞선 세부 알고리즘 구현 약속

## 4. 핵심 기술 결정

| 영역 | 표준 | 선택 이유 |
|------|------|-----------|
| 앱 | Android Kotlin | ARCore Geospatial/Streetscape 활용에 가장 현실적 |
| AR | ARCore Geospatial API | GPS 단독 방식의 오차를 보완할 핵심 수단 |
| 백엔드 | Spring Boot 3.3 / Java 17 | 현재 저장소 설정과 일치, REST API와 데이터 계층 구성에 적합 |
| DB | PostgreSQL 16 + PostGIS 3.4 | 반경 검색, 좌표/폴리곤 저장, 공간 쿼리 지원 |
| 캐시 | Redis 7 | 외부 API 호출과 점수 계산 결과 캐싱 |
| 데이터 수집 | Python 스크립트 | 공공데이터 수집/정제 작업과 분리하기 쉬움 |

## 5. 시스템 개요

```text
[Python 수집 스크립트]
        ↓
[PostgreSQL + PostGIS]
        ↓
[Spring Boot API]
        ↓
[Android 앱]
```

역할 분리 원칙:

- 데이터 수집은 `data/`에서 수행
- 서비스용 API와 비즈니스 규칙은 `backend/`에서 수행
- 화면, 지도, AR 경험은 `android/`에서 수행

## 6. 단계별 목표

| 단계 | 목표 | 산출물 |
|------|------|--------|
| Phase 1 | 주변 단지/동 탐색 흐름 확보 | 건물 목록, 기본 거래 정보, 단순 상세 진입 |
| Phase 2 | 동 상세와 지도/AR 전환 정리 | 건물 상세, 단지/동 정보, 거래 이력 |
| Phase 3 | 층 단위 표현 고도화 | 층 근사치 표현, 층 기반 UI 보강 |
| Phase 4 | 생활 인프라 점수화 | 카테고리별 점수, 프리셋, 비교 화면 |
| Phase 5 | 구미 특화 데이터 확장 | 버스/CCTV 등 지역 데이터 강화 |

현재 팀 기준에서 먼저 맞춰야 하는 것은 `Phase 1 계약 정리`와 `백엔드/앱 skeleton 위에 기능 연결`입니다.

## 7. 핵심 리스크

| 리스크 | 영향 | 현재 기준 대응 |
|--------|------|----------------|
| GPS/방향센서 오차 | 인접 동 오인 가능 | MVP는 후보 제시와 드릴다운 UI 중심으로 설계 |
| ARCore 정밀 기능 의존 | 특정 환경에서 정밀 인식 실패 | GPS/GIS 기반 폴백을 유지 |
| 동명/단지명 불일치 | 거래 데이터와 건물 데이터 매칭 실패 | 단지/동 정규화 테이블 전제 |
| 외부 API 호출 제한 | 응답 지연 및 실패 | 서버 캐싱과 사전 적재 중심으로 설계 |
| 구현 상태 과대 문서화 | 팀원 온보딩 혼선 | README에 현재 상태와 목표 범위를 분리 |
