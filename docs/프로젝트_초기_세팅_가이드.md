## 1. 프론트엔드 프레임워크 결정

### Android Kotlin (Android 전용)

| 항목 | Android Kotlin | Flutter |
|------|---------------|---------|
| ARCore Geospatial | 공식 지원 (최강) | 플러그인 불안정, Streetscape 미지원 |
| Streetscape Geometry | 완벽 지원 | 미지원 |
| Rooftop Anchor | 완벽 지원 | 미지원 |
| ML Kit OCR | 네이티브 지원 | 플러그인 경유 |
| Google Maps SDK | 완전 지원 | 제한적 |
| 공식 코드랩 | Kotlin 기준 | 없음 |
| 학습 비용 | Android Studio만 필요 | Dart + Flutter 추가 학습 |

**결론**: Android Kotlin으로 확정. ARCore의 핵심 기능(Streetscape Geometry, Rooftop Anchor)이 Android 네이티브에서만 완전 지원되므로, 12주 기간 내 목표 기능을 구현하기 위해 Android에 집중한다.

> 참고: iOS는 ARKit을 사용해야 하므로 별도 개발이 필요하다. 5명 팀으로 12주 내에 양 플랫폼을 지원하는 것은 비현실적이다.

## 2. IDE 및 개발환경 설정

### 2.1 프론트엔드 팀 (R2 강형준, R4 권희성, R5 나웅주)

**Android Studio** (최신 안정 버전, Ladybug 이상)

설치 및 설정:
1. [Android Studio 다운로드](https://developer.android.com/studio)
2. 설치 후 SDK Manager에서 설정:
   - **SDK Platforms**: Android 14 (API 34) 설치
   - **SDK Tools**: Android SDK Build-Tools, Android Emulator, Google Play services
3. 프로젝트 설정:
   - `minSdkVersion`: 28 (Android 9, ARCore 최소 요구)
   - `targetSdkVersion`: 34 (Android 14)
   - Kotlin: 1.9+
   - Gradle: 8.x
4. 필수 의존성 (build.gradle.kts):
   - ARCore SDK
   - Google Maps SDK
   - Retrofit (API 호출)
   - ML Kit OCR

> ARCore는 에뮬레이터에서 동작하지 않는다. 반드시 **ARCore 지원 실제 기기**가 필요하다.
> [지원 기기 목록](https://developers.google.com/ar/devices)

### 2.2 백엔드 팀 (R3 엄태원)

**VSCode** 권장 (PyCharm도 가능)

설치 및 설정:
1. [Python 3.11+](https://www.python.org/downloads/) 설치
2. VSCode 확장 설치:
   - Python (Microsoft)
   - Pylance
   - Black Formatter
   - REST Client (API 테스트용)
   - Docker
   - PostgreSQL (cweijan)
3. 가상환경 생성:
   ```bash
   cd backend
   python -m venv .venv
   source .venv/bin/activate  # macOS/Linux
   # .venv\Scripts\activate   # Windows
   pip install -r requirements.txt
   ```

### 2.3 PM / 데이터 수집 (R1 여창훈)

**VSCode + Python**

설치 및 설정:
1. Python 3.11+ 설치
2. VSCode + Python 확장
3. 데이터 수집 환경:
   ```bash
   cd data/collectors
   python -m venv .venv

   pip install -r requirements.txt
   ```

4. Postman 설치 (API 테스트용) - [다운로드](https://www.postman.com/downloads/)

### 2.4 공통 IDE 설정

프로젝트 루트의 `.editorconfig` 파일이 모든 IDE에서 자동 적용된다:
- UTF-8 인코딩
- LF 줄바꿈
- Kotlin/Python: 4칸 들여쓰기
- JSON/YAML: 2칸 들여쓰기

## 3. Git/GitHub 협업 전략

### 3.1 저장소 구조: 모노레포

프론트엔드, 백엔드, 데이터, 문서를 하나의 저장소에서 관리한다.

**이유**:
- 5명 소규모 팀에서 멀티레포는 관리 오버헤드가 큼
- API 명세, 문서를 한 곳에서 추적
- PR 리뷰 시 전체 변경사항 한 번에 파악 가능

### 3.2 브랜치 전략

```
main         ← 발표/시연 가능한 안정 상태. 직접 push 금지
  └── develop    ← 개발 통합 브랜치. 모든 feature는 여기로 PR
        ├── feature/frontend/arcore-setup       (R2)
        ├── feature/frontend/building-detail     (R4)
        ├── feature/frontend/map-view            (R5)
        ├── feature/backend/fastapi-init         (R3)
        ├── feature/backend/trade-api            (R3)
        ├── feature/data/apt-collector           (R1)
        └── docs/api-spec                        (공통)
```

**규칙**:
- `main`: Phase 완료 시에만 `develop`에서 머지
- `develop`: PR을 통해서만 코드 추가
- `feature/*`: 개인 작업 브랜치

### 3.3 역할별 브랜치 네이밍

| 역할 | 접두사 | 예시 |
|------|--------|------|
| R1 (PM/데이터) | `feature/data/` | `feature/data/gumi-apt-collector` |
| R2 (AR/기술리드) | `feature/frontend/` | `feature/frontend/arcore-geospatial` |
| R3 (백엔드) | `feature/backend/` | `feature/backend/trade-api` |
| R4 (상세UI) | `feature/frontend/` | `feature/frontend/building-detail-card` |
| R5 (지도/점수) | `feature/frontend/` | `feature/frontend/map-view` |
| 공통 (문서) | `docs/` | `docs/api-spec-v2` |

### 3.4 작업 흐름 (Feature Branch Workflow)

```bash
# 1. develop 브랜치에서 최신 코드 가져오기
git checkout develop
git pull origin develop

# 2. 새 feature 브랜치 생성
git checkout -b feature/backend/trade-api

# 3. 작업 및 커밋
git add .
git commit -m "feat(backend): 실거래가 API 연동 모듈 추가"

# 4. 원격에 push
git push origin feature/backend/trade-api

# 5. GitHub에서 develop으로 PR 생성
# → 리뷰어 지정 → 승인 후 머지
```

### 3.5 커밋 메시지 컨벤션

```
<타입>(<영역>): <요약>

타입:
  feat     - 새 기능
  fix      - 버그 수정
  docs     - 문서만 변경
  style    - 코드 동작 변경 없음 (포맷팅 등)
  refactor - 리팩토링
  test     - 테스트 추가/수정
  chore    - 빌드, 설정, 패키지 등

영역:
  frontend, backend, data, docs, infra

예시:
  feat(backend): 실거래가 API 연동 모듈 추가
  fix(frontend): AR 태그 위치 오프셋 보정
  docs(docs): API 명세서 v1 작성
  chore(infra): docker-compose 초기 설정
```

### 3.6 PR(Pull Request) 규칙

- 모든 코드는 PR을 통해 `develop`에 머지
- **리뷰어 규칙**:
  - 프론트엔드 PR: **R2(강형준)** 필수 리뷰
  - 백엔드 PR: **R2(강형준)** 또는 **R1(여창훈)** 리뷰
  - 데이터/문서 PR: 아무나 1명 리뷰
- 최소 **1명 승인** 후 머지
- `.env`나 API 키가 포함되어 있지 않은지 반드시 확인

### 3.7 GitHub Projects 활용

1. **Board 뷰** 생성: `To Do` → `In Progress` → `Review` → `Done`
2. **라벨** 생성:
   - 영역: `frontend`, `backend`, `data`, `docs`
   - 유형: `bug`, `enhancement`, `question`
   - 단계: `phase-1`, `phase-2`, `phase-3`, `phase-4`, `phase-5`
3. **Milestone** 5개 생성: Phase 1 MVP ~ Phase 5 마무리
4. Issue 생성 시 담당자(Assignee) 지정

### 5.5 편의시설 점수화 알고리즘

**6개 카테고리**: 의료, 교육, 생활편의, 교통, 안전, 여가

**거리별 점수**:
| 거리 | 도보 | 점수 |
|------|------|------|
| 0~300m | ~4분 | 100 |
| 300~500m | ~6분 | 90 |
| 500~800m | ~10분 | 75 |
| 800~1200m | ~15분 | 55 |
| 1200~1600m | ~20분 | 35 |
| 1600m+ | 20분+ | 15 |

**카테고리 점수** = 최근접 시설 점수 + 밀도 보너스 (반경 500m 내 시설 수 기반)

**종합 점수** = sum(카테고리 점수 x 가중치)

**4가지 가중치 프리셋**:
| 프리셋 | 의료 | 교육 | 편의 | 교통 | 안전 | 여가 |
|--------|------|------|------|------|------|------|
| 기본 균형 | 20% | 20% | 20% | 20% | 10% | 10% |
| 육아 가족 | 15% | 30% | 15% | 15% | 20% | 5% |
| 직장인 | 10% | 5% | 25% | 35% | 10% | 15% |
| 시니어 | 35% | 5% | 25% | 20% | 10% | 5% |

**등급**: S(90+) / A(75~89) / B(60~74) / C(45~59) / D(30~44) / F(~29)

## 6. 데이터 파이프라인

### 6.1 사전 적재 (배치) vs 실시간 호출

| 데이터 | 방식 | 주기 | 담당 |
|--------|------|------|------|
| 아파트 단지 목록 | 배치 적재 | 최초 1회 + 월 1회 | R1 수집 → R3 적재 |
| 단지 기본정보 | 배치 적재 | 최초 1회 + 월 1회 | R1 수집 → R3 적재 |
| 건축물대장 | 배치 적재 | 최초 1회 + 월 1회 | R1 수집 → R3 적재 |
| 실거래가 (12개월) | 배치 적재 | 일 1회 (최신 월) | R1 수집 → R3 적재 |
| 건물 폴리곤 (Vworld) | 배치 적재 | 최초 1회 + 분기 1회 | R3 |
| 편의시설 (Kakao) | 배치 적재 | 주 1회 | R3 |
| 구미시 버스/CCTV/어린이집 | 배치 적재 | 최초 1회 + 월 1회 | R1 수집 → R3 적재 |
| **역지오코딩** | **실시간** | 매 요청 | R3 (서버) |

### 6.2 API 호출 제한 대응

| API | 일일 한도 | 구미시 예상 호출량 | 여유 |
|-----|---------|-----------------|------|
| 실거래가 | 10,000건 | ~12건 (월별 x 12) | 충분 |
| 건축물대장 | 10,000건 | ~300건 (30동 x 수 페이지) | 충분 |
| 공동주택 목록 | 10,000건 | ~10건 | 충분 |
| Vworld 지오코딩 | 40,000건 | 사용자 수에 비례 | 충분 |
| Kakao Local | 30,000건 | ~900건 (150단지 x 6카테고리) | 충분 |

구미시 하나만 대상이므로 호출 제한에 걸릴 가능성은 거의 없다.

### 6.3 API 키 관리

1. **R1(여창훈)**이 API 키 발급:
   - data.go.kr: 건축물대장, 실거래가, 공동주택 단지 (3개 서비스 활용신청, 인증키 공유)
   - vworld.kr: 지오코딩 + 2D Data
   - Kakao Developers: REST API 키
2. 팀 카톡 또는 비공개 채널로 키 공유
3. 각자 `backend/.env.example`을 복사하여 `.env` 생성, 키 입력
4. **`.env`는 절대 git에 커밋하지 않는다** (.gitignore에 포함됨)

## 7. Docker 환경 구성

### 7.1 서비스 구성

| 서비스 | 이미지 | 포트 | 용도 |
|--------|-------|------|------|
| db | postgis/postgis:16-3.4 | 5432 | PostgreSQL + PostGIS |
| redis | redis:7-alpine | 6379 | 캐시 |
| backend | (빌드) | 8000 | FastAPI 서버 |

### 7.2 실행 방법

```bash
# 1. 환경변수 파일 생성
cp backend/.env.example backend/.env
# .env 파일에 실제 API 키 입력

# 2. Docker 컨테이너 실행
docker-compose up -d

# 3. 확인
# - FastAPI: http://localhost:8000/health
# - Swagger UI: http://localhost:8000/docs
# - DB 접속: psql -h localhost -U arproperty -d arproperty

# 4. 종료
docker-compose down
```

### 7.3 DB 초기화

`docker-compose up` 시 `backend/scripts/init_db.sql`이 자동 실행되어 7개 테이블 + 인덱스 + 구미시 법정동코드 초기 데이터가 생성된다.

수동 실행이 필요한 경우:
```bash
docker exec -i arproperty-db psql -U arproperty -d arproperty < backend/scripts/init_db.sql
```
