## 1. 프로젝트 개요

### 1.1 문제 정의

부동산 매수 희망자가 특정 지역을 직접 방문하여 매물을 탐색할 때, **지도 앱과 부동산 앱을 번갈아 사용해야 하는 불편함**이 존재한다. 건물의 외관, 주변 환경 등 현장 정보와 가격 데이터를 통합적으로 파악하기 어렵다.

### 1.2 해결 방안

스마트폰 카메라로 건물을 비추면 **실거래가, 전용면적, 건축년도 등의 정보가 AR(증강현실)로 건물 위에 오버레이**되는 앱을 개발한다. 국토교통부 공공데이터 API를 활용하여 신뢰성 있는 실거래가 정보를 제공하며, **주변 생활 편의시설의 거리·밀도를 점수화**하여 해당 위치의 생활 편의성까지 한눈에 보여준다.

### 1.3 한 문장 정의

> **구미시에서 사용자가 스마트폰 카메라로 아파트·건물을 비추면, 해당 건물의 단지명·동 정보·최근 실거래가·건축 정보·주변 생활 인프라 점수를 AR과 지도에서 동시에 보여주는 현장형 부동산 탐색 서비스**

### 1.4 주요 기능

| 기능 | 설명 |
|------|------|
| AR 실거래가 오버레이 | GPS + 카메라 방향센서로 건물 인식, AR 태그 표시 |
| 상세 거래이력 조회 | AR 태그 탭 → 거래 내역 리스트, 가격 변동 그래프 |
| 지도 기반 탐색 | 2D 지도 위 마커 표시, AR/지도 뷰 전환 |
| 필터링 및 검색 | 면적, 가격, 건축년도, 거래 유형 조건 필터링 |
| 가격 변동 시각화 | 꺾은선 그래프, 지역 평균 대비 비교 |
| 건축물대장 연동 | 건축년도, 층수, 구조, 승강기 수, 연면적 등 표시 |
| **생활 편의시설 점수** | 병원·학교·편의점·교통 등 편의시설 거리 기반 종합 점수 |

### 1.5 핵심 데이터 파이프라인

```
[사용자 위치 + 카메라 방향]
        │
```

## 2. 주제 정의 및 차별점

### 2.1 이 프로젝트의 본질

이 프로젝트는 "부동산 앱"이라기보다 **공간 인식 기반 부동산 정보 시각화 시스템**에 가깝다. 단순히 "AR을 붙인 부동산 앱"이 아니라, **현장에서 눈으로 보고 있는 건물 자체를 인터페이스로 바꾸는 현장형 임장(현장답사) 도우미**이다.

### 2.2 제목 후보

1. **구미시 AR 실거래가 브라우저**
2. **현장형 AR 부동산 임장 도우미**
3. **아파트 건물 인식 기반 구미 부동산 정보 서비스**
4. **구미시 공공데이터 기반 AR 부동산 시각화 앱**

### 2.3 이 주제가 강한 이유

- **검색형이 아니라 시야형(scene-first)** 이다.
- **공공데이터 기반이라 신뢰성 설명이 쉽다.**
- **구미시라는 구체적 지역**을 잡으면 범위 관리가 된다.
- 기술적으로는 **단지/동/층**까지 확장 가능한 계단형 구조를 제시할 수 있다.
- **생활 편의시설 점수화**로 단순 가격 정보를 넘어 종합 주거 가치를 제공한다.

### 2.4 핵심 차별화 포인트

```
┌─────────────────────────────────────────────────┐
│           기존 앱: 검색 → 리스트 → 클릭           │
│                (간접적, 단계적)                    │
│                                                   │
│    ▼▼▼  본 프로젝트: 카메라로 비추면 즉시 확인  ▼▼▼   │
│              (직관적, 실시간, 현장형)               │
└─────────────────────────────────────────────────┘
```

1. **AR 기반 현장 탐색**: 국내 유일하게 카메라로 건물을 비추면 실거래가가 오버레이되는 앱
2. **공공데이터 기반 신뢰성**: 허위매물 문제 없이 국토교통부 실거래가만 활용
3. **한눈에 비교**: 여러 건물이 동시에 보일 때 각각의 가격을 즉시 비교 가능
4. **생활 편의성 점수**: 가격뿐 아니라 주변 편의시설 접근성까지 종합 평가
5. **구미 지역 특화**: 구미시 아파트 데이터에 최적화된 서비스

## 3. 왜 구미시인가

### 3.1 프로젝트 관점에서 구미시가 좋은 이유

- 전국 단위 서비스보다 **범위를 명확히 줄여서** 실현 가능성을 보여주기 좋다.
- 국가 공공데이터 API는 전국 공통 구조라서 **구미시 코드만 넣으면 바로 조회 가능**하다.
- 구미시는 아파트 단지, 구축/신축 주거지, 산업단지 배후 생활권이 섞여 있어 **파일럿 도시**로 설명하기 좋다.

### 3.2 구미시 핵심 코드

| 구분 | 코드 |
|------|------|
| 시도코드 | `47` (경상북도) |
| 시군구코드 / LAWD_CD | `47190` (구미시) |
| 좌표 범위 (경도) | 128.20 ~ 128.50°E |
| 좌표 범위 (위도) | 36.05 ~ 36.25°N |
| 중심점 (구미시청 부근) | 128.3445, 36.1195 |

구미시 주요 법정동코드(10자리, 앞 5자리 = 47190)는 행정표준코드관리시스템(https://www.code.go.kr/stdcode/regCodeL.do )에서 "경상북도 구미시"로 검색하여 확인 가능하다.

### 3.3 파일럿 권역 선정 기준

처음부터 구미시 전역을 다루기보다, 아래 기준으로 **1~2개 권역**을 먼저 잡는 것이 좋다.

| 선정 기준 | 이유 |
|---|---|
| 아파트 밀집도 | 여러 동이 동시에 보이는 장면이 필요 |
| 도로 시야 확보 | 카메라가 외벽/동번호를 잘 잡아야 함 |
| GPS·나침반 안정성 | 고층 밀집 지역일수록 오차가 커질 수 있음 |
| Street View/VPS 가능성 | ARCore Geospatial 실험 가능 여부 확인 |
| 구축/신축 혼합 여부 | 비교 스토리텔링이 쉬움 |
| 생활 인프라 다양성 | 편의시설 점수 기능 검증에 유리 |

### 3.4 예시 파일럿 후보 (현장답사 전제)

- **송정동·형곡동권**: 구축 아파트와 도심 생활권 비교에 유리
- **옥계동·인동동권**: 아파트 밀집과 생활 인프라 연결 설명에 유리
- **산동읍권**: 비교적 신축·확장 주거지 사례 설명에 유리

> 발표에서는 "구미시 전체가 아니라 **구미시 내 1개 생활권부터 검증하는 파일럿**"이라고 잡는 것이 가장 현실적이다.

## 4. 공공데이터 API 상세 조사

두 가지 주요 데이터 포털에서 API 키를 발급받아야 한다:
1. **공공데이터포털** (data.go.kr) - 실거래가, 건축물대장, 공동주택 정보 API용

## 5. 데이터 레이어 정리 및 우선순위

이 프로젝트는 4개 축으로 구성된다:
1. **가격 데이터**: 실거래가 / 전월세
2. **건물 데이터**: 건축물대장 / 건물 폴리곤 / 단지 식별정보
3. **생활권 데이터**: 버스, CCTV, 어린이집 등
4. **편의시설 데이터**: 병원, 약국, 편의점, 학교 등 (점수화용)

### 5.1 1순위: 실거래가 레이어

**A. 국토교통부 아파트 매매 실거래가 자료**
- 역할: AR 화면의 핵심 가격 숫자 제공
- 활용: 대표 실거래가 태그, 면적별 가격 비교, 층별 거래 분포

**B. 국토교통부 아파트 전월세 실거래가 자료**
- 역할: 전세/월세 확장
- 활용: 전세가율, 월세 흐름, 실거주 판단 도구

### 5.2 1순위: 건물 인식 레이어

**C. 건축물대장정보 서비스**
- 역할: 건물의 물리적 속성 확보
- 핵심 필드: 지상층수, 지하층수, 구조, 연면적, 사용승인일, 세대수
- 활용: 층수 계산, 건물 정보 카드, 노후도, 저층부 필로티/상가 보정

**D. 브이월드 2D 데이터 API / WFS / Geocoder**
- 역할: 좌표↔주소 변환, 주변 건물 폴리곤 조회

**E. GIS건물통합정보 (WMS/WFS / 파일데이터)**
- 역할: 건물 공간정보 + 건축물대장 속성의 통합 기준 레이어
- 활용: 건물 폴리곤 기준 DB 구축, 아파트 동별 중심좌표/면/높이 모델링

## 7. 건물 인식 기술 조사

### 7.1 GPS + 방향센서 기반 건물 식별

**원리**: GPS 좌표 + 자기장/자이로스코프 방위각 → "사용자가 어디서 어느 방향을 보고 있는가"

```
1. 사용자 GPS 좌표 획득
2. 반경 R(예: 500m) 내의 건물 목록 조회
3. 각 건물의 GPS 좌표와 사용자 위치 간의 방위각(Bearing) 계산
4. 카메라 방향(기기 방위각)과 각 건물의 방위각을 비교
5. 화면 중앙 방향과 가장 가까운 방위각의 건물을 추정
```

**정확도**

| 요소 | 정확도 | 비고 |
|------|--------|------|
| GPS (실외, 개활지) | 3~5m | 최적 조건 |
| GPS (도심, 고층건물) | 10~30m | 멀티패스 오류 |
| 방위각 (Magnetometer) | ±5~15도 | 금속/전자기기 간섭 시 저하 |
| 자이로스코프 보정 후 | ±2~5도 | 센서 퓨전 적용 시 |

- **치명적 한계**: 아파트 밀집 환경에서 GPS 오차로 인접 동 구분이 어려움

### 7.2 Computer Vision 기반 건물 인식

| 방식 | 설명 | 부동산 앱 적합성 |
|------|------|-----------------|
| Google Cloud Vision API | 랜드마크 인식, 유명 건물 높은 인식률 | **일반 아파트 인식 불가** |
| OpenCV 특징점 매칭 | SIFT/SURF/ORB로 건물 외관 특징점 추출 | 전국 아파트 DB화 비현실적 |
| Google Lens | ML + 컴퓨터 비전 + GPS | 일반 아파트 동 식별 부적합 |

### 7.3 ARCore Geospatial API (핵심 기술)

**VPS(Visual Positioning System)**: Google Street View의 수백억 장 이미지 기반 시각적 위치 측위. GPS보다 **센티미터 수준** 정밀도. **87개국 이상** 지원 (한국 포함).

**Streetscape Geometry API (가장 유망한 기술)**
- 반경 **100m 이내** 건물/지형 3D 메시 데이터 실시간 제공
- **LOD1**: 건물 풋프린트 수직 압출 모델
- **LOD2**: 실제 건물 형태에 가까운 고해상도 메시
- 건물 메시에 대한 **Hit-Test** 지원 (RaycastStreetscapeGeometry)
- 건물 표면에 **Anchor 부착** 가능
- **Rooftop Anchor**: 건물 옥상에 AR 콘텐츠 배치 가능

**Geospatial Depth API**: 최대 **65m** 거리의 건물 깊이 정보 제공, 오클루전 처리 가능

**한계점**
- Streetscape Geometry는 **건물 단위** 메시만 제공 (층/세대 구분 없음)
- Street View 커버리지가 없는 지역에서 작동 불가
- 한국 내 아파트 단지 내부 커버리지 확인 필요

### 7.4 ARKit (iOS)

| 기능 | 건물 인식 활용도 |
|------|-----------------|
| ARGeoTracking | 중간 (도시 제한적) |

- Google Streetscape Geometry에 대응하는 건물 메시 API가 없음

### 7.5 LiDAR 센서

- ARCore Geospatial Depth와 결합 시 **근거리 깊이 보정**에 활용 가능

### 7.6 건물 인식 접근법 비교 결론

| 방식 | 정확도 | 구현 난이도 | 채택 여부 |

## 8. 아파트 동/층 식별 기술

### 8.1 "아파트를 인식한다"의 단계적 분해

| 인식 대상 | 예시 | 현실성 |
|---|---|---|
| 단지 인식 | "여기는 OO아파트다" | 높음 |
| 동 인식 | "지금 보이는 건 101동이다" | 중~높음 |
| 층 인식 | "지금 가리키는 지점은 15층 부근이다" | 중간 |
| 호수 인식 | "1502호다" | 낮음 |

> **과제 범위 목표: 단지 + 동 + 대략적 층. 호수 식별은 권장하지 않음.**

### 8.2 아파트 동 구분 방법

#### GPS + 건물 좌표 매칭
- 각 동은 고유한 GPS 좌표(건물 중심점)를 가짐
- GIS건물통합정보에서 각 동별 건물 중심 좌표 확보

#### ARCore Geospatial API 활용
- Streetscape Geometry 건물 메시에 고유 ID 부여
- 메시와 GIS 건물 데이터 좌표 매칭 → 동 식별
- Rooftop Anchor로 동별 라벨링

#### 동번호 OCR
- 아파트 외벽의 `101`, `102` 같은 큰 숫자를 OCR로 인식
- 공간 매칭 + OCR이 동시에 맞으면 신뢰도 상승
- OCR 실패해도 기본 공간 매칭이 남아 서비스 유지

### 8.3 층 구분 문제 - 핵심 난제

```
┌──────────────────────────────────┐
│  25층  │ 전용 84㎡  │  8.5억     │
│  24층  │ 전용 84㎡  │  8.3억     │
│  ...   │           │            │
│   3층  │ 전용 84㎡  │  7.2억     │
│   1층  │ 전용 84㎡  │  6.5억     │
│════════════════════════════════│
│   같은 건물인데 층마다 가격이 다르다  │
└──────────────────────────────────┘
```

**왜 어려운가?**
- GPS는 수평 위치만 제공 (어느 **층**을 보고 있는지는 모름)
- 기압 센서는 **사용자 자신의 층**만 감지 (카메라가 가리키는 건물의 층 감지 불가)

### 8.4 층 식별 기술적 해결 방안

#### 방안 A: 건물 높이 기반 층 추정 (계산 기반)

```python
건물_총높이 = Streetscape_Geometry_메시_높이
총_층수 = 건축물대장_API.지상층수              # 예: 25층
층_높이 = 건물_총높이 / 총_층수               # 예: 75m / 25 = 3m

# AR에서 사용자가 탭한 지점의 Y좌표로 층 추정
탭_높이 = AR_hit_test_결과.y좌표 - 건물_바닥_좌표.y
추정_층 = math.ceil(탭_높이 / 층_높이)        # 예: 45m → 15층
```

#### 방안 B: 드릴다운 UI (UX 기반 해결) — MVP 권장

```
[AR 화면] 건물에 대표 태그 표시
    │  사용자가 태그 탭
    ▼
[단지 상세 화면] 아파트명, 동 선택
    │  동 선택
    ▼
[층별 거래 정보] 층별 실거래가 리스트
    │  특정 층 선택
    ▼
[상세 정보] 거래금액, 전용면적, 계약일, 가격 추이 그래프
```

#### 방안 C: AR 층별 히트맵 오버레이 (고도화)

건물 표면에 층별 색상 그라데이션: 빨강(고가)~초록(저가)

#### 층 인식 단계적 전략

| 단계 | 층 처리 방식 |
|------|-------------|
| **Phase 1 (MVP)** | AR 태그에 최근 거래가 범위만 표시, 탭 시 드릴다운 UI |
| **Phase 2** | 건물 높이/층수 기반 층 추정 + 해당 층 근처 가격 하이라이트 |
| **Phase 3** | 건물 표면에 층별 가격 히트맵 오버레이 |

|------|------|

### 10.3 거리 기반 점수 산출 알고리즘

#### Step 1: 편의시설 검색

건물 중심 좌표를 기준으로 **반경 2km 이내**의 편의시설을 카테고리별로 조회한다.

#### Step 2: 거리별 점수 부여

도보 이동 시간을 기준으로 점수를 산정한다 (보행 속도 약 80m/분 기준).

| 거리 구간 | 도보 시간 | 점수 |
|-----------|-----------|------|
| 0 ~ 300m | 약 4분 이내 | 100점 |
| 300 ~ 500m | 약 4~6분 | 90점 |
| 500 ~ 800m | 약 6~10분 | 75점 |
| 800 ~ 1,200m | 약 10~15분 | 55점 |
| 1,200 ~ 1,600m | 약 15~20분 | 35점 |
| 1,600m 초과 | 20분 초과 | 15점 |
| 반경 내 해당 시설 없음 | - | 0점 |

#### Step 3: 카테고리 내 복수 시설 처리

같은 카테고리에 시설이 여러 개 있으면, **가장 가까운 시설의 점수를 주 점수**로 하되 **밀도 보너스**를 부여한다.

```
카테고리_점수 = 최근접_시설_점수 + 밀도_보너스

밀도_보너스 = min(10, (반경 500m 내 시설 수 - 1) × 3)
```

예시: 500m 내 편의점이 4개 → 최근접 100점 + 밀도 보너스 min(10, 3×3) = 9점 → **카테고리 점수 = 100 (상한 100)**

#### Step 4: 종합 점수 계산

```
종합_점수 = Σ (카테고리_점수 × 카테고리_가중치)

종합_점수 = (의료_점수 × 0.20) + (교육_점수 × 0.20) + (생활편의_점수 × 0.20)
          + (교통_점수 × 0.20) + (안전_점수 × 0.10) + (여가_점수 × 0.10)
```

#### Step 5: 등급 분류

| 종합 점수 | 등급 | 라벨 |
|-----------|------|------|
| 90 ~ 100 | S | 최우수 |
| 75 ~ 89 | A | 우수 |
| 60 ~ 74 | B | 양호 |
| 45 ~ 59 | C | 보통 |
| 30 ~ 44 | D | 미흡 |
| 0 ~ 29 | F | 부족 |

### 10.4 점수 산출 의사코드

```python
import math

def haversine_distance(lat1, lon1, lat2, lon2):
    """두 좌표 간 직선 거리(m) 계산"""
    R = 6371000  # 지구 반경(m)
    phi1, phi2 = math.radians(lat1), math.radians(lat2)
    dphi = math.radians(lat2 - lat1)
    dlam = math.radians(lon2 - lon1)
    a = math.sin(dphi/2)**2 + math.cos(phi1)*math.cos(phi2)*math.sin(dlam/2)**2
    return 2 * R * math.atan2(math.sqrt(a), math.sqrt(1-a))

def distance_to_score(distance_m):
    """거리(m)를 점수(0~100)로 변환"""
    if distance_m <= 300:
        return 100
    elif distance_m <= 500:
        return 90
    elif distance_m <= 800:
        return 75
    elif distance_m <= 1200:
        return 55
    elif distance_m <= 1600:
        return 35
    elif distance_m <= 2000:
        return 15
    else:
        return 0

def density_bonus(count_within_500m):
    """500m 반경 내 시설 수에 따른 밀도 보너스"""
    return min(10, max(0, (count_within_500m - 1)) * 3)

def calculate_category_score(building_coord, facilities):
    """
    건물 좌표와 해당 카테고리 시설 목록으로 카테고리 점수 계산
    building_coord: (lat, lon)
    facilities: [(lat, lon, name), ...]
    """
    if not facilities:
        return 0, []
    
    distances = []
    for fac in facilities:
        dist = haversine_distance(building_coord[0], building_coord[1], fac[0], fac[1])
        distances.append((dist, fac[2]))
    
    distances.sort(key=lambda x: x[0])
    
    nearest_score = distance_to_score(distances[0][0])
    count_500m = sum(1 for d, _ in distances if d <= 500)
    bonus = density_bonus(count_500m)
    
    category_score = min(100, nearest_score + bonus)
    
    return category_score, distances[:5]  # 점수 + 가까운 5개 시설 목록

def calculate_livability_score(building_coord, all_facilities, weights):
    """
    종합 생활 편의 점수 계산
    
    all_facilities: {
        "medical": [(lat, lon, name), ...],
        "education": [...],
        "convenience": [...],
        "transport": [...],
        "safety": [...],
        "leisure": [...]
    }
    
    weights: {
        "medical": 0.20,
        "education": 0.20,
        "convenience": 0.20,
        "transport": 0.20,
        "safety": 0.10,
        "leisure": 0.10
    }
    """
    results = {}
    total_score = 0
    
    for category, facilities in all_facilities.items():
        score, nearest_list = calculate_category_score(building_coord, facilities)
        results[category] = {
            "score": score,
            "nearest": nearest_list
        }
        total_score += score * weights.get(category, 0)
    
    total_score = round(total_score, 1)
    
    # 등급 분류
    if total_score >= 90: grade = "S"
    elif total_score >= 75: grade = "A"
    elif total_score >= 60: grade = "B"
    elif total_score >= 45: grade = "C"
    elif total_score >= 30: grade = "D"
    else: grade = "F"
    
    return {
        "total_score": total_score,
        "grade": grade,
        "categories": results
    }
```

| 데이터 유형 | 캐시 주기 | 저장 방식 |
|------------|-----------|-----------|
| 구미시 버스 정류장 위치 | 월 1회 갱신 | PostGIS DB |
| 구미시 CCTV 위치 | 월 1회 갱신 | PostGIS DB |

| 건물별 종합 점수 | 주 1회 재계산 | Redis 캐시 |

**PostGIS 기반 거리 쿼리 예시**
```sql
-- 특정 건물 좌표에서 반경 2km 내 병원 조회 (거리순)
SELECT name, category,
       ST_Distance(
           point_geom::geography,
           ST_SetSRID(ST_MakePoint(128.3445, 36.1195), 4326)::geography
       ) AS distance_m
FROM living_infra_gumi
WHERE category = 'medical'
  AND ST_DWithin(
      point_geom::geography,
      ST_SetSRID(ST_MakePoint(128.3445, 36.1195), 4326)::geography,
      2000  -- 2km
  )
ORDER BY distance_m
LIMIT 10;
```

## 11. 하이브리드 건물 인식 알고리즘

### 11.1 추천 인식 스택

```text
[사용자 GPS]
    +
[자이로/나침반/방향센서]
    +
[주변 건물 폴리곤 DB]
    +
[단지/동 메타데이터]
    +

    +
[동번호 OCR]
    =
[현재 바라보는 단지/동/층 후보 추정]
```

> **아파트 인식은 "사진 분류" 문제가 아니라 "공간 매칭" 문제에 가깝다.**

### 11.2 알고리즘 흐름

**Step 1: 주변 건물 후보 추출**
- 사용자 위치 기준 반경 150~300m 내 건물 폴리곤 조회
- 아파트/공동주택 계열만 우선 필터링
- 단지/동 메타데이터와 조인

**Step 2: 카메라 방향으로 1차 필터링**
- 사용자 방위각과 각 건물 중심점의 bearing 계산
- 화면 시야각(FOV) 안에 들어오는 건물만 남김

**Step 3: 화면 중앙선 또는 탭 ray와 가장 가까운 건물 선택**

**Step 4: 동번호 OCR로 확인**
- 카메라 프레임에서 큰 숫자를 OCR
- 후보 건물의 동번호와 매칭

**Step 5: 신뢰도 점수 계산**
- 거리 점수 + 시야 중심선 일치도 + 폴리곤/메시 일치도 + OCR 숫자 일치 + ARCore tracking 품질

**Step 6: 확신이 낮으면 후보 2~3개 제시**
- 점수가 높으면 자동 선택
- 점수가 낮으면 `101동 / 103동 / 105동 중 어떤 건물인지 선택` 후보 제시

> **틀리게 확정하는 것보다, 후보를 좁혀 보여주는 UX가 더 낫다.**

### 11.3 건물 메시 ↔ GIS 데이터 매칭 의사코드

```python
def match_building(streetscape_mesh, gis_buildings):
    """
    Streetscape Geometry의 건물 메시 중심좌표를
    GIS건물통합정보의 건물 중심좌표와 매칭
    """
    mesh_center = streetscape_mesh.get_center_coordinate()
    
    best_match = None
    min_distance = float('inf')
    
    for building in gis_buildings:
        distance = haversine(mesh_center, building.coordinate)
        if distance < min_distance and distance < THRESHOLD:  # 예: 20m
            min_distance = distance
            best_match = building
    
    return best_match  # 단지명, 동명, 층수, 높이 등 포함

def estimate_floor(hit_point_y, building_info):
    """
    Ray-cast 히트포인트의 Y좌표로 층 추정
    """
    ground_elevation = building_info.ground_elevation
    total_height = building_info.total_height
    total_floors = building_info.total_floors
    
    floor_height = total_height / total_floors
    estimated_floor = int((hit_point_y - ground_elevation) / floor_height) + 1
    
    return max(1, min(estimated_floor, total_floors))
```

- 지도 보기 버튼

#### Level 3: 층별 보기

- 세로형 층 바, 저층/중층/고층 구분
- 최근 거래가 있는 층 하이라이트
- 층 선택 시 상세 거래 이력

#### Level 4: 생활환경 카드

- 편의시설 카테고리별 상세 (거리, 시설명)
- 버스 정거장 수, CCTV 밀도
- 어린이집/학교 정보
- 입주 예정 단지
- 가중치 프리셋 전환

> **AR은 요약**, **상세는 카드/리스트/지도**로 넘기는 구조가 가장 좋다.

### 12.2 AR 뷰 ↔ 지도 뷰 전환

두 뷰를 자유롭게 전환할 수 있으며, 각 뷰에서 편의시설 정보를 다르게 시각화한다:
- **AR 뷰**: 건물 태그에 종합 점수 포함, 건물 비교
- **지도 뷰**: 편의시설 아이콘 레이어 토글, 거리선 시각화, 히트맵

## 13. 단계별 구현 전략

### Phase 1: MVP (2~3주)

```
목표: 주변 단지/동 인식 + 대표 시세 오버레이

✦ GPS + 역지오코딩 + 방위각으로 주변 아파트 단지 식별
✦ 카메라 화면에 단지별 대표 가격 AR 태그 표시
✦ 태그 탭 → 드릴다운 UI로 층별 거래 내역 조회
✦ 2D 지도 뷰 ↔ AR 뷰 전환
```

### Phase 2: 핵심 기능 (3~4주)

```
목표: 개별 동 단위 식별 + 상세 정보

✦ ARCore Geospatial API 적용 → 개별 동 단위 식별
✦ 건물 메시 표면에 AR 앵커로 태그 고정
✦ 가격 변동 그래프, 건축물대장 정보 상세 화면
✦ 필터링 기능 (면적, 가격, 건축년도)
```

### Phase 3: 층별 매핑 고도화 (4~5주)

```
목표: 층 근사치 표시

✦ 건물 높이/층수 기반 층별 매핑
✦ 층별 가격 히트맵 오버레이
✦ 지역 평균 대비 비교 그래프
✦ 성능 최적화 (데이터 캐싱, 거리 기반 LOD)
```

### Phase 4: 생활 편의시설 점수화 (3~4주)

```
목표: 편의시설 점수 산출 + 비교 기능

✦ 편의시설 데이터 수집 파이프라인 구축 (Kakao Local + 구미시 데이터)
✦ PostGIS 기반 거리 계산 및 점수 산출 엔진
✦ AR 태그에 편의시설 종합 점수 표시
✦ 상세 화면: 카테고리별 분석, 가까운 시설 목록
✦ 지도 뷰 편의시설 레이어 (아이콘 + 거리선)
✦ 가중치 프리셋 전환 (기본 균형 / 육아 가족 / 직장인 / 시니어)
✦ 건물 간 편의시설 점수 비교표
```

### Phase 5: 구미 지역 특화 (2~3주)

```
목표: 구미 지역 특화 데이터 확장

✦ 버스·정류장 데이터 연동 (교통 카테고리 강화)
✦ CCTV 밀도 기반 안전 점수 정교화
✦ 공급 예정 단지 표시
✦ 생활권 비교 (권역별 편의시설 평균 점수)
```

## 부록: 교수님 질문 대응용 핵심 문장

1. **이 프로젝트는 검색형 부동산 앱이 아니라, 현장에서 건물을 보며 정보를 바로 이해하는 AR 임장 도우미입니다.**
2. **핵심 구현 목표는 호수가 아니라 단지/동/층 수준입니다.**
3. **아파트 인식은 이미지 분류보다 공간 매칭 문제이므로, GPS·건물 폴리곤·단지 메타데이터·OCR을 함께 쓰는 하이브리드 방식이 현실적입니다.**
4. **층 인식은 건물 높이와 총 층수, 사용자가 가리킨 벽면 hit-point를 이용해 근사 추정하는 방식이 가장 설득력 있습니다.**
5. **생활 편의시설 점수화 시스템을 통해 단순 가격 정보를 넘어, 의료·교육·교통·안전 등 6개 카테고리의 생활 편의성을 종합 평가하여 사용자의 의사결정을 돕습니다.**
6. **구미시를 대상으로 시작하면 공공데이터와 현장 검증 범위를 관리하기 좋고, 이후 타 지역으로 확장 가능합니다.**
